The Framework

What is SCQCS?

Schr√∂dinger's Cat Quantum Cryptography & Security

SCQCS is a set of architectural patterns and security principles for building systems that need both privacy and accountability. Named after the famous thought experiment, it embodies a core insight: data should remain sealed until deliberately observed‚Äîand observation should leave auditable evidence.

The framework provides guidance for implementing append-only event chains, sealed vault storage, controlled "break-glass" access, and cryptographic agility. It's designed for systems where you need to prove what happened without enabling surveillance.

SCQCS is not a product‚Äîit's a philosophy and pattern library. Implementations may vary in how they apply these principles to their specific domains.

Principles

Designed to age well

These aren't features to be added‚Äîthey're constraints that shape every decision.

01

Witness, not watcher

Support evidence without enabling surveillance. Default to data minimization. Resist scope creep and retroactive expansion.

data-minimization purpose-limitation
02

Audit over trust

Prefer verifiable integrity over promises. Cryptographic proofs are more durable than policies, contracts, or good intentions.

tamper-evident verifiable
03

Exceptions leave scars

Break-glass access may be necessary‚Äîbut it should be hard to trigger casually and designed to prevent silent invocation.

least-privilege attribution
04

Plan for rotation

Algorithms weaken. Keys get compromised. Design for migration from day one. Crypto agility isn't optional.

crypto-agility key-rotation
05

Local-first when possible

External dependencies are attack surface. Prefer architectures that survive without vendor clouds and require explicit egress.

local-first explicit-egress
06

Assume adversarial insiders

The threat model includes administrators, developers, and operators. Make misuse architecturally difficult, not just policy-prohibited.

abuse-resistance defense-in-depth

Access Patterns

Break-glass with receipts

Emergency access is sometimes necessary. SCQCS defines patterns that make it deliberate, scoped, and durably recorded.

üîì

Accountable exception handling

Three properties every break-glass implementation should enforce

01

Scoped by default

Access should be constrained to minimum necessary scope‚Äîtime range, record types, and purpose.

02

Multi-party when appropriate

Consider requiring multiple approvals or split keys. Single-actor access is a design choice, not a default.

03

Durable attribution

Every access creates a tamper-evident receipt: who, when, justification, scope, and outcomes.
